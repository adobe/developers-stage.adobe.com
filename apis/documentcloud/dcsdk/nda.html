Sales proposal

Overview

As is often the case in business, there is a need to complete a non-disclosure agreement to protect the confidentiality of a party. Often, the information that needs to be submitted into an NDA such as the names of the parties involved, addresses, etc. are already known information in a database or may be provided through a web interface. The challenge developers face is that dynamically generating documents with custom data and sending them out for signatures has been a manual process. It often requires custom branded documents to be populated by end-users with data first before they can be sent out for signatures.

APIs Used

  • Adobe Document Generation API, a Microsoft Word add-in
  • Adobe Document Generation Tagger
  • Adobe Sign

End User Scenario

A vendor manager has the information of the vendor he would like to do business with including the name of the contact, the address of the business, as well as the scope of the NDA. The vendor manager goes onto an online web portal to initiate the NDA process. At the click of a button, the data is collected from the database, passed to generate the NDA document, and then sent to Adobe Sign to send for signature.

Document Flow

In the scenario where there needs to be negotiation between the two parties, the developer will use Document Generation API to provide a document back in Word format, which would allow for redlining in Microsoft Word. Adobe Document Generation Tagger, a Microsoft Word add-in, can be used by the developer or the vendor manager to quickly build a custom branded NDA template for the company to standardize their agreements for accuracy. Using this custom template with dynamic text tags, JSON data can be sent with the template to Document Generation API then immediately sent to Adobe Sign for signature and countersignature.
